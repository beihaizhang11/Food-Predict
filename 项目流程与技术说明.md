# 项目流程与技术说明（浅显版）

## 1. 这个项目在做什么

一句话：  
我们把餐饮点评数据（文本+评分+时间）变成一个“可解释的预测系统”。

系统最终要回答两个问题：
1. 未来评分会怎么变（LSTM 预测）？
2. 为什么会这样变（知识图谱 + 属性/菜品驱动解释）？

---

## 2. 全流程总览（从数据到前端）

完整链路是：

1. 生成/导入点评数据  
2. NLP 分析点评文本（情感、实体、关键词）  
3. 构建知识图谱（店铺-菜品-属性关系）  
4. 做时间序列预测（LSTM）  
5. 把结果在前端联动展示（趋势 + 图谱 + 工作流指标）

---

## 3. 每一步做了什么（输入、输出、技术、为什么）

## 步骤 A：数据准备与导入

- 输入：
  - `data/mock_reviews.json`（或真实 JSON）
  - 每条包含：`shop_id`、`dish`、`rating`、`review_text`、`review_time`、`tags`
- 处理：
  - 校验字段
  - 写入 MySQL `reviews`
  - 对重复 `id` 做覆盖更新
- 输出：
  - MySQL 结构化评论数据
- 技术栈：
  - Python、Flask、SQLAlchemy、MySQL
- 为什么这样选：
  - MySQL 对结构化数据检索和聚合稳定，做分页、筛选、时间查询很方便。

---

## 步骤 B：NLP 文本分析

- 输入：
  - `review_text`（评论文本）
- 处理：
  - 文本清洗
  - SBERT 生成语义向量（embedding）
  - 情感模型打分（正/负）
  - NER + 规则抽取属性（口味、服务、价格、环境）
- 输出：
  - `sentiment`（情感分）
  - `entities`（实体）
  - `keywords`（关键词）
  - `embedding`
  - 并回写 MySQL（`sentiment` + `analysis_cache`）
- 技术栈：
  - `sentence-transformers`（SBERT）
  - `transformers`（情感模型、NER）
- 为什么这样选：
  - 预训练模型上手快、效果稳定，适合毕业设计先跑通完整链路。

---

## 步骤 C：知识图谱构建

- 输入：
  - 评论结构化字段（用户、店铺、菜品、评分、时间）
  - NLP 抽取出的属性实体
- 处理：
  - 写入 Neo4j 节点：`User`、`Shop`、`Dish`、`Attribute`
  - 写入关系：
    - `(User)-[:RATED]->(Dish)`
    - `(Shop)-[:OFFERS]->(Dish)`
    - `(Dish)-[:HAS_ATTRIBUTE]->(Attribute)`
- 输出：
  - 可查询的图谱结构
  - 图谱聚合视图（前端默认展示）
- 技术栈：
  - Neo4j + Cypher
- 为什么这样选：
  - 图关系天然适合“解释原因”：能直观看到哪家店、哪道菜、哪些属性在影响评分。

---

## 步骤 D：LSTM 趋势预测

- 输入：
  - 按月/周聚合后的时间序列特征：
    - 平均评分 `rating`
    - 平均情感 `sentiment`
    - 评论量 `review_count`
- 处理：
  - 归一化
  - 构造滑动窗口（lookback）
  - 训练 LSTM
  - 预测未来若干期评分
  - 输出误差指标（MAE、RMSE）
- 输出：
  - `history`（历史趋势）
  - `forecast`（未来趋势）
  - `metrics`（误差）
- 技术栈：
  - PyTorch（LSTM）
  - pandas / numpy / scikit-learn
- 为什么这样选：
  - LSTM 对序列建模直观，和“趋势预测”题目贴合，解释和答辩都比较好讲。

---

## 步骤 E：预测解释（把图谱和预测串起来）

- 输入：
  - 预测结果（历史 + 未来）
  - 评论原始数据（最近窗口与上一窗口）
  - 标签/菜品统计
- 处理：
  - 计算“下一期变化量”
  - 找出正向驱动菜品、负向驱动菜品
  - 统计高频属性（如口味、服务）
- 输出：
  - 可读解释文本（summary）
  - 驱动因子列表（top positive / negative dishes, top attributes）
- 技术栈：
  - Python 数据分析逻辑（pandas）
- 为什么这样选：
  - 解决“只预测不解释”的问题，让图谱和预测形成业务闭环。

---

## 步骤 F：前端联动展示

- 输入：
  - `/api/predict`：趋势 + 解释
  - `/api/graph`：图谱聚合结构
  - `/api/workflow`：流程步骤 + 指标快照
- 处理：
  - 趋势折线图
  - 图谱关系图
  - 工作流步骤 + 指标卡片
  - 筛选条件联动（时间、店铺、菜品）
- 输出：
  - 一页可演示界面：既看未来趋势，也看原因与内部处理状态
- 技术栈：
  - 原生 HTML/CSS/JS
  - ECharts、Cytoscape.js
- 为什么这样选：
  - 开发效率高、依赖轻、适合毕业设计快速展示完整结果。

---

## 4. 接口与模块对应关系（便于讲解）

- `POST /api/import`：导入 + NLP + 图谱增量更新
- `GET /api/reviews`：结构化点评查询
- `GET /api/graph`：图谱查询（支持 summary/detail）
- `POST /api/predict`：LSTM 预测 + 原因解释
- `GET /api/workflow`：工作流步骤 + 过程指标

---

## 5. 技术栈总表（答辩可直接用）

- 后端：Flask
- 关系库：MySQL
- 图数据库：Neo4j
- NLP：Transformers + Sentence-Transformers
- 时序预测：PyTorch LSTM
- 数据处理：pandas / numpy / scikit-learn
- 前端：HTML/CSS/JS + ECharts + Cytoscape.js

---

## 6. 为什么这套方案适合毕业设计

1. 完整：覆盖“数据 -> 算法 -> 可视化 -> 解释”全链路。  
2. 可演示：前端可直接展示预测结果和原因，不是只看模型指标。  
3. 可扩展：后续可替换更强模型（如 Transformer 时序模型）或接入真实平台数据。  
4. 可复现：有 mock 数据、脚本、接口，老师现场也能复跑。

---

## 7. 一句话总结（答辩开场可用）

这个系统不是单纯做“评分预测”，而是把预测结果和知识图谱解释结合起来，让用户同时看到“会发生什么”和“为什么会发生”。
